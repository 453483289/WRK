427 //
428 // This defines the WMI Guid object. There are really 5 different type of
429 // objects that are created from this data structure:
430 //
431 // QuerySet Object returned to the data consumer to allow them to
432 // send queries and sets to a particular set of providers. It
433 // does not use a flag.
434 //
435 // Notification Object is used by data consumers to receive events. It
436 // is created when a data consumer wants to receive an event for a
437 // particular guid and queues up the received events until they are
438 // retrieved by the consumer. When the object is deleted it is
439 // removed from the list of objects maintained by a GuidEntry and
440 // a event disable request is sent to the devices that expose
441 // the events if this is the last object open to the event.
442 // These have no flag set
443 //
444 // Request Object is created on behalf of a user mode data provider
445 // when it registers its guids with WMI. It acts like a Notification
446 // object in that requests (in the form of events) are received
447 // by it and then can be picked up by the user mode creator. It uses
448 // the WMIGUID_FLAG_REQUEST_OBJECT flag. When a Request Object is
449 // being deleted it will clean up any guids registered for it and
450 // send a reply message to any reply objects waiting to receive
451 // a message from it.
452 //
453 // Reply Object is created as a product of Creating a user mode logger.
454 // This object acts as a notification object to deliver replies
455 // sent by request objects. When it is closed it will remove itself
456 // from any request list that it may be part of and clear the
457 // reference to it in any request objects and deref itself to account
458 // for that. These use the WMIGUID_FLAG_REPLY_OBJECT flag
459 //
460 // Security Objects are created so that the security apis can have
461 // a mechanism to change the security descriptors for the guids
462 // These have the WMIGUID_FLAG_SECURITY_OBJECT
