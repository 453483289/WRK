13 Abstract:
14 
15  The Exception filter is used by the file system and cache manager
16  to handle error recovery. The basic idea is to have the top level
17  file system entry points (i.e., the FSD entry points and FSP dispatch
18  loop) have a try-except around their code, and then whenever the
19  file system or cache manager reach an error case they raise an
20  appropriate status. Then the exception handler catches the exception
21  and can either complete the request, send it off to the fsp, verify the
22  volume, or bugcheck. We only bugcheck if the raised exception is
23  unexpected (i.e., unhandled).
24 
25  This module provides two routines for filtering out exceptions. The
26  first routine is used to normalize status values to be one of the
27  value handled by the filter. That way if we get an exception not handled
28  by the filter then we know that the system is in real trouble and we
29  just bugcheck the machine. The second routine is used to ask if
30  a status value is within the set of values handled by the filter.
31 
32  The value of status handled by this filter are listed in the routine
33  FsRtlIsNtstatusExpected.
34 
35 --*/

310 /*++

312 Routine Description:
313 
314  This routine is given an NTSTATUS value and make a determination as to
315  if this value indicates that the complete device has failed and therefore
316  should no longer be used, or if the failure is one that indicates that
317  continued use of the device is ok (i.e. a sector failure).
318 
319 Arguments:
320 
321  Status - the NTSTATUS value to test.
322 
323 Return Value:
324 
325  TRUE - The status value given is believed to be a fatal device error.
326  FALSE - The status value given is believed to be a sector failure, but not
327  a complete device failure.
328 --*/
329 
330 {
331  if (NT_SUCCESS(Status)) {
332 
333  //

335  //
336 
337  return FALSE;
338  }
339 
340  switch (Status) {
341  case STATUS_CRC_ERROR:
