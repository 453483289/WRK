155 //
156 // Set a throttle for charging a given write against the total number of dirty
157 // pages in the system, for the purpose of seeing when we should invoke write
158 // throttling.
159 //
160 // This must be the same as the throttle used for seeing when we must flush
161 // temporary files in the lazy writer. On the back of the envelope, here
162 // is why:
163 //
164 // RDP = Regular File Dirty Pages
165 // TDP = Temporary File Dirty Pages
166 // CWT = Charged Write Throttle
167 // -> the maximum we will charge a user with when we see if
168 // he should be throttled
169 // TWT = Temporary Write Throttle
170 // -> if we can't write this many pages, we must write temp data
171 // DPT = Dirty Page Threshold
172 // -> the limit when write throttling kicks in
173 //
174 // PTD = Pages To Dirty
175 // CDP = Charged Dirty Pages
176 //
177 // Now, CDP = Min( PTD, CWT).
178 //
179 // Excluding other effects, we throttle when:
180 // #0 (RDP + TDP) + CPD >= DPT
181 //
182 // To write temporary data, we must cause:
183 // #1 (RDP + TDP) + TWT >= DPT
184 //
185 // To release the throttle, we must eventually cause:
186 // #2 (RDP + TDP) + CDP < DPT
187 //
188 // Now, imagine TDP >> RDP (perhaps RDP == 0) and CDP == CWT for a particular
189 // throttled write.
190 //
191 // If CWT > TWT, as we drive RDP to zero (we never defer writing regular
192 // data except for hotspots or other very temporary conditions), it is clear
193 // that we may never trigger the writing of temporary data (#1) but also
194 // never release the throttle (#2). Simply, we would be willing to charge
195 // for more dirty pages than we would be willing to guarantee are available
196 // to dirty. Hence, potential deadlock.
197 //
198 // CWT < TWT I leave aside for the moment. This would mean we try not to
199 // allow temporary data to accumulate to the point that writes throttle as
200 // a result. Perhaps this would even be better than CWT == TWT.
201 //
202 // It is legitimate to ask if throttling temporary data writes should be relaxed
203 // if we see a large amount of dirty temp data accumulate (and it would be very
204 // easy to keep track of this). I don't claim to know the best answer to this,
205 // but for now the attempt to avoid temporary data writes at all costs still
206 // fits the reasonable operation mix, and we will only penalize the outside
207 // oddcase with a little more throttle/release.
208 //
209 
210 #define WRITE_CHARGE_THRESHOLD (64 * PAGE_SIZE)

1158 /*++
1159 
1160 Routine Description:
1161 
1162  This routine checks to see if we meet the conditions to force this
1163  cached write to be write through even though the file object was not
1164  opened for write through access. We will force the operation to write
1165  through when the following conditions are met:
1166 
1167  1. The write originated from a remote user (i.e., the write is coming
1168  through SRV.SYS)
1169 
1170  2. This cached write is large enough that we would otherwise throttle
1171  the write because we have too much dirty data in the cache right now.
1172 
1173 Arguments:
1174 
1175  FileObject - The file object for this cached write.
1176 
1177  WriteLength - The amount of data being written
1178 
1179  SharedCacheMap - SharedCacheMap for this file.
1180 
1181  SetForcedWriteThroughFlag - TRUE if the caller is going to convert the
1182  current write to write through based on the return value of this
1183  function. If so, we will set the FORCE_WRITE_THROUGH flag in the
1184  shared cache map now. FALSE if the caller just wants to know if we are
1185  under the conditions that would cause a write to be forced write through.
1186 
1187 Return Value:
1188 
1189  Returns TRUE if the write should be forced to be write through, or FALSE
1190  if we can go ahead with a regular cached write.
1191 
1192 --*/
1193 
1194 {
1195  BOOLEAN ForcedWriteThrough = FALSE;
1196  KIRQL OldIrql;
1197 
1198  //
1199  // If this file is from SRV, we don't want to throttle writes on this
1200  // machine because these cached writes were already throttled on the client.
1201  // But, we can't just allow SRVs cached writes to go unchecked because
1202  // it can fill up all the memory in the machine. So, if we would otherwise
1203  // throttle this cached write, we will force the write to WRITE_THROUGH
1204  // so that we don't end up generating more dirty data in the cache.
1205  //
1206  // It is very important to set the FORCED_WRITE_THROUGH flag on the
1207  // shared cache map so that the lazy writer thread will still send the
1208  // VDL updates to the file system if these writes ended up extending VDL.
1209  //
1210 
1211  if (IoIsFileOriginRemote(FileObject) &&
1212  !CcCanIWrite( FileObject,
1213  WriteLength,
1214  FALSE,
1215  MAXUCHAR - 2 )) {
1216 
1217  ForcedWriteThrough = TRUE;
1218 
1219  if (SetForceWriteThroughFlag &&
1220  !FlagOn(SharedCacheMap->Flags, FORCED_WRITE_THROUGH)) {
1221 
1222  CcAcquireMasterLock( &OldIrql );
1223  SetFlag(SharedCacheMap->Flags, FORCED_WRITE_THROUGH);
1224  CcReleaseMasterLock( OldIrql );
1225  }
1226  }
1227 
1228  return ForcedWriteThrough;
