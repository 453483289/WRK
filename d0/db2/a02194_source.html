75 // --- HCELL_INDEX ---
76 //
77 //
78 // Handle to a cell -> effectively the "virtual" address of the cell,
79 // HvMapCell converts this to a "real" address, that is, a memory
80 // address. Mapping scheme is very much like that standard two level
81 // page table. No mappings stored in file, they are built up when
82 // the file is read in. (The INDEX in HCELL_INDEX is historical)
83 //
84 // Bit 31 30-21 20-12 11-0
85 // +----------------------------+
86 // | T | Table | Block | Offset |
87 // +----------------------------+
88 //
89 // T = Type(1)= 0 for stable ("normal") storage
90 // 1 for volatile storage
91 //
92 // Table(10) = Index into directory of mapping tables, selects a table.
93 // Each mapping table is an array of HMAP_ENTRY structures.
94 //
95 // Block(9) = Index into Table, selects an HMAP_ENTRY. HMAP_ENTRY
96 // contains address of area in memory that this HCELL_INDEX
97 // maps to. (Base of memory copy of Block)
98 //
99 // Offset(12) = Offset within page, of the Cell header for the cell
100 // of interest.
101 //

215 //
216 // ===== On Disk Structures =====
217 //
218 
219 //
220 // NOTE: Hive storage is always allocated in units of 4K. This size
221 // must be used on all systems, regardless of page size, since
222 // the file format needs to be transportable among systems.
223 //
224 // NOTE: The integrity code depends on certain blocks (e.g., the
225 // BASE block) being at least as large as the size of a physical
226 // sector. (Otherwise data that should be left alone will
227 // be written because the FS has to block/deblock.) This means
228 // that the current code will not work with sectors > 4K.
229 //
230 // NOTE: A hive on disk always contains at least two blocks of storage.
231 // 1 block for the base block, and 1 for the minimum 1 bin.
232 //
233 // NOTE: Only modified parts of the hive get written to disk.
234 // This is not just for efficiency, but also to avoid risk
235 // of destruction of unlogged data. Dirty bits keep track
236 // of what has been modified, they reside in a simple
237 // bit map attached to the hive. One bit for each logical
238 // sector of 512 bytes.
239 //
240 // If the physical sector size of the machine is less than 512,
241 // no matter, we'll always write in clumps of 512. If the
242 // physical sector size is greater than 512, we'll always clump
243 // data together so that we log and write data
244 // in chunks of that size. Physical sector sizes > 4K will
245 // not work correctly (logging will not work right, so system
246 // crashes may lose data that would not otherwise be lost.)
247 //
248 
249 
250 //
251 // An on disk image of a hive looks like this:
252 //
253 // +---------------------------------------+
254 // | HBASE_BLOCK | 1 Hive Block == 4K
255 // | |
256 // +---------------------------------------+ <- HBLOCK_SIZE boundary
257 // | Bin - 1 to N 4K blocks |
258 // | Each contains a signature, size, and |
259 // | a boundary tag heap internal to |
260 // | itself. Once allocated lives forever |
261 // | and always at same file offset. |
262 // +---------------------------------------+ <- HBLOCK_SIZE boundary
263 // | Bin ... |
264 // +---------------------------------------+ <- HBLOCK_SIZE boundary
265 // ...
266 // +---------------------------------------+ <- HBLOCK_SIZE boundary
267 // | Last allocated Bin, new bins are put |
268 // | immediately after this one. |
269 // +---------------------------------------+ <- HBLOCK_SIZE boundary
270 //
271 // Hive files must allocate on HBLOCK_SIZE boundaries because they
272 // might be written on many different systems, and must therefore be
273 // set up for the largest cluster size we will support.
274 //
275 
276 //
277 // The log file format is:
278 //
279 // +-------------------------------+
280 // | HBASE_BLOCK copy |
281 // +-------------------------------+ <- cluster (usually 512) bound
282 // | DirtyVector |
283 // | (length computed from length |
284 // | in the base block |
285 // | (with "DIRT" on front as a |
286 // | signature) |
287 // +-------------------------------+ <- cluster (usually 512) bound
288 // | Dirty Data |
289 // +-------------------------------+ <- cluster (usually 512) bound
290 // | Dirty Data |
291 // +-------------------------------+ <- cluster (usually 512) bound
292 // | ... |
293 // +-------------------------------+
294 //
295 // Recovery consists of reading the file in, computing which clusters
296 // of data are present from the dirtyvector, and where they belong in
297 // the hive address space. Position in file is by sequential count.
298 //
299 // Logs can allocate on cluster boundaries (physical sector size of
300 // host machine) because they will never be written on any machine other
301 // than the one that created them.
302 //
303 // For log to be valid:
304 //
305 // Signature, format, major.minor must match expected values.
306 // Sequence1 and Sequence2 must match.
307 // CheckSum must be correct.
308 // Signature on DirtyVector must be correct
309 //
310 // For log to be applicable:
311 //
312 // Sequence in log must match sequence in hive.
313 // TimeStamp in log must match TimeStamp in hive.
314 // Hive must be in mid-update state, or have bogus header.
315 //
316 
317 //
318 // --- HBASE_BLOCK --- on disk description of the hive
319 //
320 
321 //
322 // NOTE: HBASE_BLOCK must be >= the size of physical sector,
323 // or integrity assumptions will be violated, and crash
324 // recovery may not work.
325 //

378 //
379 // ===== In Memory Structures =====
380 //
381 
382 //
383 // In memory image of a Hive looks just like the on-disk image,
384 // EXCEPT that the HBIN structures can be spread throughout memory
385 // rather than packed together.
386 //
387 // To find an HCELL in memory, a mechanism that takes an HCELL_INDEX and
388 // derives a memory address from it is used. That mechanism is very
389 // similar to a two level hardware paging table.
390 //
391 // A bit map is used to remember which parts of the hive are dirty.
392 //
393 // An HBLOCK can be in three different states
394 // 1. Present in memory. BlockAddress and BinAddress are valid pointers.
395 // This is the normal state of an HBLOCK.
396 //
397 // 2. Discardable. The HBIN containing this HBLOCK is completely free, but
398 // the bin is dirty and needs to be written to the hive file before it
399 // can be free. This is the state we will be in if somebody frees a
400 // cell, causing the entire HBIN to become free. HvpEnlistFreeCell will
401 // transition all the HBLOCKs in the free HBIN to this state, but will
402 // not free their memory. After the dirty HBLOCKs are flushed to the
403 // file, the memory will be freed.
404 //
405 // Note that if we need to allocate more storage from an HBIN in this
406 // state, HvAllocateCell will simply change its state back to State 1
407 // and it will be usable.
408 //
409 // An HBLOCK in this state has a valid BlockAddress and BinAddress, but
410 // the HMAP_DISCARDABLE bit will be set.
411 //
412 // 3. Discarded. The HBIN containing this HBLOCK is completely free, and
413 // is not dirty (i.e. it is marked as free in the hive file as well).
414 // There is no memory allocated to contain this HBIN. After HvSyncHive
415 // writes out an HBIN that is in State 2, it frees its pool and the
416 // HBIN moves into this state.
417 //
418 // In order to use this HBIN, memory must be allocated to back it, and
419 // the HBIN and initial HCELL must be recreated. (we could re-read it
420 // from the hive file, but there's not much point in that since we know
421 // that it is entirely free, so we might as well just recreate it and
422 // save the disk i/o)
423 //
424 // An HBLOCK in this state has a NULL BlockAddress in the map.
425 // The BinAddress will contain the next HCELL in the free list, so
426 // we can reconstruct this when we need it.
427 // The HMAP_NEWALLOC bit will be set for the first HBLOCK in the HBIN.
428 //
429

592 //
593 // --- HHIVE --- In memory descriptor for a hive.
594 //
595 
596 //
597 // HHIVE contains pointers to service procedures, and pointers to
598 // map structure.
599 //
600 // NOTE: Optimization - If the size of a hive is less than what can
601 // be mapped with a single HMAP_TABLE (HTABLE_SLOTS * HBLOCK_SIZE,
602 // or 2 megabytes) there is no real HMAP_DIRECTORY. Instead,
603 // a single DWORD in the HHIVE acts as the 0th entry of the
604 // directory.
605 //
606 // NOTE: Free Storage Management - When a hive is loaded, we build up
607 // a display (vector) of lists of free cells. The first part
608 // of this vector contains lists that only hold one size cell.
609 // The size of cell on the list is HCELL_PAD * (ListIndex+1)
610 // There are 15 of these lists, so all free cells between 8 and
611 // 120 bytes are on these lists.
612 //
613 // The second part of this vector contains lists that hold more
614 // than one size cell. Each size bucket is twice the previous
615 // size. There are 8 of these lists, so all free cells between 136 and
616 // 32768 bytes are on these lists.
617 //
618 // The last list in this vector contains all cells too large to
619 // fit in any previous list.
620 //
621 // Example: All free cells of size 1 HCELL_PAD (8 bytes)
622 // are on the list at offset 0 in FreeDisplay.
623 //
624 // All free cells of size 15 HCELL_PAD (120 bytes)
625 // are on the list at offset 0xe.
626 //
627 // All free cells of size 16-31 HCELL_PAD (128-248 bytes)
628 // are on the list at offset 0xf
629 //
630 // All free cells of size 32-63 HCELL_PAD (256-506 bytes)
631 // are on the list at offset 0x10.
632 //
633 // All free cells of size 2048 HCELL_PAD (16384 bytes)
634 // OR greater, are on the list at offset 0x17.
635 //
636 // FreeSummary is a bit vector, with a bit set to true for each
637 // entry in FreeDisplay that is not empty.
638 //
639
