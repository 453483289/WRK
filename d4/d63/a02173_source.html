89 //
90 // The registry is a large data structure that has had poor locality.
91 // To improve performance without changing the on disk structure, we
92 // cache the frequently used registry data to minimize reference on
93 // registry data.
94 //
95 // A KCB (Key Control Block) is the core structure for registry cache.
96 // It uses HashValue for quick cache lookup and contains the most
97 // frequently used data in a key node.
98 //
99 // It contains the most frequently used data in a key node:
100 // Security, Flags, and Value index.
101 //
102 // A KCB may also contains additional information
103 // (which are cached lazily) about its subkeys, value nodes and values' data.
104 //
105 // The subkey information is distinquished by ExtFlags. See CM_KCB_* below.
106 // The value nodes and data are distinguished by a bit in the variable.
107 // See CMP_IS_CELL_CACHED.
108 //
109 // Caches for value data will be created during query process, the cached
110 // structure is shown as the following picture. The structure is almost
111 // the same as the registry structure
112 // except they are pointers to the allocation instead of offset index on hive.
113 //
114 // To minimize the name string storage space KCB's, we do not store the complete
115 // path name of the key in the kcb, instead, we implemented the tree structure
116 // (like the registry hive structure) to share name prefix.
117 // Also, knowing that there are lots of keys sharing same names,
118 // we create NameBlock structure so KCB's of same names
119 // can share the NameBlock. NameBlock is compressed.
120 //
121 // Meanings when the following bits are set in ExtFlags:
122 // 1. The following bits are used for Parse and are for
123 // non-symbolic keys. Also, at most one bit can be set at any given time.
124 // CM_KCB_KEY_NON_EXIST : This key is a fake key (no such key in the hive).
125 // CM_KCB_NO_SUBKEY : This key is has no subkey.
126 // CM_KCB_SUBKEY_ONE : This key has only one subkey and IndexHint is
127 // the first four characters of this subkey.
128 // CM_KCB_SUBKEY_HINT : This key has the first four characters of all
129 // its subkeys (buffer pointed by IndexHint).
130 //
131 // 2. CM_KCB_SYM_LINK_FOUND: This bit is only for symbolic keys. It
132 // indicates that the symbolic link has been
133 // resolved and the KCB for the link is pointed to
134 // by ValueCache.RealKcb.
135 // In this case, the Value Index of this key is no longer
136 // available in the KCB. (We hardly query the value
137 // of a symbolic link key other than finding the path
138 // of the real key anyway).
139 //
140 // 3. CM_KCB_NO_DELAY_CLOSE: This bit is only used for non-symbolic keys and is
141 // independent of bits on item 1. When set, it indicates that
142 // key should not be kept in delay close when the reference
143 // count goes to zero.
144 // This is for the case when a key has no open handles but
145 // still has subkeys in the cache.
146 // When its last subkey is kicked out of cache, we do not
147 // want to keep this key around.
148 // This is done so CmpSearchForOpenSubKeysInCachen can clean
149 // up the cache properly before a key can be unloaded.
150 //
151 //
152 // KCB
153 // +-------------------+
154 // | ... | (Typical case)
155 // +-------------------+ Value Index
156 // | ValueCache | +-->+---------+ Value Key (with small data)
157 // + +----------------+ | | o--------->+-----------+
158 // | | ValueList o---+ +---------+ | .... |
159 // | +---- Union -----| | | +-----------+
160 // | | RealKcb o---+ +---------+ | Data (S) |
161 // | +----------------| | | | +-----------+
162 // | | | +---------+
163 // | | | | |
164 // | | | +---------+ Value Key (with large data)
165 // | | | | o--------->+-----------+
166 // | | | +---------+ | ... |
167 // | | | | | +-----------+
168 // | | | +---------+ | Data (L) o------+
169 // | | | +-----------+ |
170 // | | | | | <---+ (Append at the end of Value Node)
171 // | | | | |
172 // | | | | |
173 // | | | +-----------+
174 // | | |
175 // | | | KCB (Symbolic link key, CM_KCB_SYM_LINK_FOUND set).
176 // | | +-->+---------+
177 // | | | |
178 // | | | |
179 // | | | |
180 // | | | |
181 // | | | |
182 // | | +---------+
183 // | |
184 // | ... |
185 // +-------------------+ Index Hint
186 // | IndexHint o------>+---------+
187 // +-------------------+ | 4 char |
188 // | | +---------+
189 // | | | 4 char |
190 // +-------------------+ +---------+
191 // | | (CM_KCB_SUBKEY_HINT)
192 // | |
193 // | |
194 // +-------------------+ Name Block
195 // | NameBlock o----------------->+----------+
196 // +-------------------+ | |
197 // +----------+
198 //
199 //
200 // The TotalLevels is used for quick comparison for notification and cache lookup.
201 //
202 // *** MP Synchronization ***
203 // The KCB lock is held for any write to KCB unless the registry is locked exclusively.
204 // KCB is also locked while reading fields that can be modified by another thread
205 // during a read operation, i.e., when the registry lock is held shared.
206 //
207 // The fields are the follows: ExtFlags, ValueCache, IndexInfo, IndexHint, or NameHint.
208 //
209 // Reading of other entries in the KCB does not need to hold the KCB lock since
210 // these entries will not change for any registry read operation. When there
211 // are changes to these entries, registry must be locked exclusively.
212 //
213 // NOTE: the KCB size is 56 bytes now, plus the pool header of 8 bytes,
214 // it fits into a 64byte allocation. Think carefully if you want to
215 // enlarge the data structure. Also, watch it if the pool allocation code changes.
216 //
217 // The RefCount in KCB is the number of open handles plus the number of cached subkeys.
218 // We can change this by having a RefCount and a CachedSubKeyCount. To not grow the
219 // structure size, we can merge the boolean Delete into ExtFlags.

303 //
304 // ----- CM_KEY_INDEX -----
305 //
306 // A leaf index may be one of two types. The "old" CM_KEY_INDEX type is used for
307 // hives circa NT3.1, 3.5, and 3.51. NT4.0 introduces the newer CM_KEY_FAST_INDEX
308 // which is used for all leaf indexes that have less than CM_MAX_FAST_INDEX leaves.
309 //
310 // The main advantage of the fast index is that the first four characters of the
311 // names are stored within the index itself. This almost always saves us from having
312 // to fault in a number of unnecessary pages when searching for a given key.
313 //
314 // The main disadvantage is that each subkey requires twice as much storage. One dword
315 // for the HCELL_INDEX and one dword to hold the first four characters of the subkey
316 // name. If one of the first four characters in the subkey name is a unicode character
317 // where the high byte is non-zero, the actual subkey must be examined to determine the
318 // name.
319 //
320 // Hive version 1 & 2 do not support the fast index. Version 3 adds support for the
321 // fast index. All hives that are newly created on a V3-capable system are therefore
322 // unreadable on V1 & 2 systems.
323 //
324 // N.B. There is code in cmindex.c that relies on the Signature and Count fields of
325 // CM_KEY_INDEX and CM_KEY_FAST_INDEX being at the same offset in the structure!
326 
327 #define INVALID_INDEX 0x80000000 // index is not valid
328 
329 #define UseFastIndex(Hive) ((Hive)->Version >= 3)
330 #define UseHashIndex(Hive) ((Hive)->Version >= HSYS_WHISTLER)
