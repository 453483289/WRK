37 NTSTATUS
38 NtAcceptConnectPort (
39  __out PHANDLE PortHandle,
40  __in_opt PVOID PortContext,
41  __in PPORT_MESSAGE ConnectionRequest,
42  __in BOOLEAN AcceptConnection,
43  __inout_opt PPORT_VIEW ServerView,
44  __out_opt PREMOTE_PORT_VIEW ClientView

46 
47 /*++
48 
49 Routine Description:
50 
51  A server process can accept or reject a client connection request
52  using the NtAcceptConnectPort service.
53 
54  The ConnectionRequest parameter must specify a connection request
55  returned by a previous call to the NtListenPort service. This
56  service will either complete the connection if the AcceptConnection
57  parameter is TRUE, or reject the connection request if the
58  AcceptConnection parameter is FALSE.
59 
60  In either case, the contents of the data portion of the connection
61  request is the data to return to the caller of NtConnectPort.
62 
63  If the connection request is accepted, then two communication port
64  objects will be created and connected together. One will be
65  inserted in the client process' handle table and returned to the
66  client via the PortHandle parameter it specified on the
67  NtConnectPort service. The other will be inserted in the server
68  process' handle table and returned via the PortHandle parameter
69  specified on the NtCompleteConnectPort service. In addition the
70  two communication ports (client and server) will be linked together.
71 
72  If the connection request is accepted, and the ServerView parameter
73  was specified, then the section handle is examined. If it is valid,
74  then the portion of the section described by the SectionOffset and
75  ViewSize fields will be mapped into both the client and server
76  process address spaces. The address in server's address space will
77  be returned in the ViewBase field. The address in the client's
78  address space will be returned in the ViewRemoteBase field. The
79  actual offset and size used to map the section will be returned in
80  the SectionOffset and ViewSize fields.
81 
82  Communication port objects are temporary objects that have no names
83  and cannot be inherited. When either the client or server process
84  calls the !f NtClose service for a communication port, the port will
85  be deleted since there can never be more than one outstanding handle
86  for each communication port. The port object type specific delete
87  procedure will then be invoked. This delete procedure will examine
88  the communication port, and if it is connected to another
89  communication port, it will queue an LPC_PORT_CLOSED datagram to
90  that port's message queue. This will allow both the client and
91  server processes to notice when a port becomes disconnected, either
92  because of an explicit call to NtClose or an implicit call due to
93  process termination. In addition, the delete procedure will scan
94  the message queue of the port being closed and for each message
95  still in the queue, it will return an ERROR_PORT_CLOSED status to
96  any thread that is waiting for a reply to the message.
97 
98 Arguments:
99 
100  PortHandle - A pointer to a variable that will receive the server
101  communication port object handle value.
102 
103  PortContext - An uninterpreted pointer that is stored in the
104  server communication port. This pointer is returned whenever
105  a message is received for this port.
106 
107  ConnectionRequest - A pointer to a structure that describes the
108  connection request being accepted or rejected:
109 
110  The ConnectionRequest structure
111 
112  ULONG Length - Specifies the size of this data structure in
113  bytes.
114 
115  CLIENT_ID ClientId - Specifies a structure that contains the
116  client identifier (CLIENT_ID) of the thread that sent the
117  request.
118 
119  The ClientId Structure
120 
121  ULONG UniqueProcessId - A unique value for each process
122  in the system.
123 
124  ULONG UniqueThreadId - A unique value for each thread in the
125  system.
126 
127  ULONG MessageId - A unique value that identifies the connection
128  request being completed.
129 
130  ULONG PortAttributes - This field has no meaning for this service.
131 
132  ULONG ClientViewSize - This field has no meaning for this service.
133 
134  AcceptConnection - Specifies a boolean value which indicates where
135  the connection request is being accepted or rejected. A value
136  of TRUE means that the connection request is accepted and a
137  server communication port handle will be created and connected
138  to the client's communication port handle. A value of FALSE
139  means that the connection request is not accepted.
140 
141  ServerView - A pointer to a structure that specifies the section that
142  the server process will use to send messages back to the client
143  process connected to this port.
144 
145  The ServerView Structure
146 
147  ULONG Length - Specifies the size of this data structure in
148  bytes.
149 
150  HANDLE SectionHandle - Specifies an open handle to a section
151  object.
152 
153  ULONG SectionOffset - Specifies a field that will receive the
154  actual offset, in bytes, from the start of the section. The
155  initial value of this parameter specifies the byte offset
156  within the section that the client's view is based. The
157  value is rounded down to the next host page size boundary.
158 
159  ULONG ViewSize - Specifies the size of the view, in bytes.
160 
161  PVOID ViewBase - Specifies a field that will receive the base
162  address of the port memory in the server's address space.
163 
164  PVOID ViewRemoteBase - Specifies a field that will receive
165  the base address of the server port's memory in the client's
166  address space. Used to generate pointers that are
167  meaningful to the client.
168 
169  ClientView - An optional pointer to a structure that will receive
170  information about the client process' view in the server's
171  address space. The server process can use this information
172  to validate pointers it receives from the client process.
173 
174  The ClientView Structure
175 
176  ULONG Length - Specifies the size of this data structure in
177  bytes.
178 
179  PVOID ViewBase - Specifies a field that will receive the base
180  address of the client port's memory in the server's address
181  space.
182 
183  ULONG ViewSize - Specifies a field that will receive the
184  size, in bytes, of the client's view in the server's address
185  space. If this field is zero, then client has no view in
186  the server's address space.
187 
188 Return Value:
189 
190  NTSTATUS - An appropriate status value.
191 
192 --*/

761  ClientView->ViewSize = ConnectMsg->ClientView.ViewSize;
762  }
763

765

769  }
770

791  }

814  // thread is woken up with a call to Complete Connect Request

818

835  //
